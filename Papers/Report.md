## Введение

**Актуальность темы.** Проектные институты России, в частности АО «Метрогипротранс», в ходе своей многолетней деятельности накопили огромные бумажные архивы проектной документации. В рамках государственной программы цифровизации эти архивы проходят сканирование, что порождает новую проблему: процесс оцифровки создает графические копии документов, но не структурированные, машиночитаемые данные. В результате тысячи файлов хранятся под техническими именами (`scan_001.jpg`), что делает поиск необходимой информации практически невозможным без ручного просмотра и обработки. Ручной ввод метаданных со штампов чертежей — монотонная, трудоемкая задача, занимающая до 30% рабочего времени сотрудников и подверженная человеческим ошибкам. Автоматизация этого процесса является высокоактуальной задачей, позволяющей значительно повысить эффективность работы с электронными архивами, снизить трудозатраты и минимизировать количество ошибок.

**Цель и задачи практики.**

**Целью** данной технологической практики является разработка прототипа программного конвейера (`gost-ocr`) для автоматического извлечения ключевых метаданных из основных надписей (штампов) отсканированных архивных чертежей.

Для достижения поставленной цели были определены следующие **задачи**:

1.  Провести анализ проблематики ручной обработки архивной документации и изучить соответствующую нормативно-техническую базу (ГОСТы на оформление проектной документации).
2.  Определить границы проекта, технологический стек и разработать архитектуру программного конвейера.
3.  Реализовать модуль предобработки изображений, включающий коррекцию поворота и перекоса.
4.  Реализовать модуль локализации основной надписи (штампа) на изображении с использованием методов компьютерного зрения.
5.  Реализовать модуль извлечения текстовых данных из области штампа и их структурирования.
6.  Обеспечить возможность пакетной обработки изображений и вывода результатов в машиночитаемом формате (JSON).
7.  Подготовить техническую документацию, описывающую функциональность и структуру проекта.

## Основная часть

### Раздел 1. Проектная часть

#### 1.1. Постановка и детализация проекта

Проект **GOST-OCR** является инициативной разработкой, нацеленной на оптимизацию внутренних бизнес-процессов в организациях с крупными архивами технической документации, таких как АО «Метрогипротранс». Отправной точкой для проекта послужил глубокий анализ (Deep Research) текущих процедур работы с отсканированными чертежами, выявивший значительные неэффективности.

**Анализ текущего бизнес-процесса («As-Is»)**

На сегодняшний день процесс обработки входящего скана чертежа в архивном отделе выглядит следующим образом:
1.  **Получение данных:** Сотрудник (архивариус) получает партию отсканированных документов в виде графических файлов (JPEG, PNG, TIFF).
2.  **Ручная обработка:** Для каждого файла сотрудник выполняет серию монотонных операций:
    *   Открывает изображение в графическом редакторе.
    *   Визуально находит основную надпись (штамп).
    *   Считывает и вручную перепечатывает ключевые метаданные (шифр проекта, номер листа, наименование и т.д.) во временный документ или напрямую в реестр (часто в MS Excel).
    *   Переименовывает файл в соответствии с извлеченными данными (например, `МГТ-2024-ПЗ_Лист_01.jpg`).
3.  **Завершение:** После обработки всей партии данные из реестра могут переноситься в систему электронного документооборота (СЭД).

Этот процесс обладает рядом существенных недостатков:
*   **Высокие трудозатраты:** На обработку одного документа уходит от 3 до 5 минут, что приводит к дням монотонной работы при больших объемах.
*   **Человеческий фактор:** Монотонность и усталость ведут к ошибкам — опечаткам в шифре, неверно указанному номеру листа.
*   **Низкая скорость:** Процесс не масштабируется. Увеличение объема архива прямо пропорционально увеличивает нагрузку на персонал.

**Предлагаемый бизнес-процесс («To-Be»)**

Проект `gost-ocr` предлагает внедрение автоматизированного конвейера, который кардинально изменяет этот процесс:
1.  **Загрузка данных:** Пользователь (архивариус) помещает всю партию отсканированных файлов в специально выделенную «входящую» папку на сервере.
2.  **Автоматическая обработка:** Программный комплекс `gost-ocr`, работающий в фоновом режиме, автоматически обнаруживает новые файлы и запускает конвейер обработки для каждого из них.
3.  **Генерация результатов:** По завершении обработки для каждого исходного изображения система создает JSON-файл, содержащий все распознанные текстовые блоки, их координаты и степень уверенности. Исходный файл может быть автоматически переименован.
4.  **Верификация:** Сотрудник открывает итоговый структурированный файл (например, единый отчет в формате MS Excel, сгенерированный на основе JSON-файлов) и быстро проверяет (верифицирует) распознанные данные, корректируя только те поля, где система показала низкую уверенность.

**Метрики для оценки эффективности**

Эффективность внедрения предлагаемого решения можно оценить по следующим ключевым показателям (KPI):
*   **Среднее время обработки одного документа:** Ожидается снижение с 3-5 минут до менее чем 30 секунд (включая автоматическую обработку и быструю верификацию).
*   **Процент ошибок при вводе данных:** Снижение ошибок, связанных с человеческим фактором, более чем на 90%. Ошибки будут возможны только на этапе верификации, а не на этапе первичного ввода.
*   **Стоимость обработки партии документов:** Прямое снижение трудозатрат (человеко-часов), необходимых для обработки стандартной партии из 500 листов, с 3-5 рабочих дней до нескольких часов.
*   **Время поиска документа в электронном архиве:** Благодаря корректным и стандартизированным именам файлов и наличию структурированного реестра, время поиска сократится с десятков минут (при ручном просмотре) до нескольких секунд (при использовании поиска по файловой системе или СЭД).

#### 1.2. Организационная структура и участники проекта

Несмотря на то, что АО «Метрогипротранс» является крупной инженерной организацией, в ней может отсутствовать выделенный IT-отдел, занимающийся проактивной разработкой подобных инструментов. Проект `gost-ocr` является инициативой, предложенной «снизу вверх». Для его успешного пилотного внедрения и дальнейшего развития предлагается следующая гипотетическая организационная структура.

**Заинтересованные стороны (Stakeholders):**

*   **Спонсор проекта:** Руководитель, наиболее заинтересованный в успехе проекта. Вероятно, это **Начальник архивного отдела** или **Главный инженер проекта (ГИП)**, который осознает ценность быстрого доступа к архивным данным и неэффективность текущих процессов.
*   **Конечные пользователи:** **Сотрудники архивного отдела, инженеры-проектировщики**, которые непосредственно работают с документацией и будут являться основными пользователями системы.
*   **Владелец продукта / Инициатор:** **Разработчик проекта (студент-практикант)**, который выполняет анализ, разработку и предлагает решение.

**Роли и зоны ответственности в рамках пилотного проекта:**

Даже если все роли выполняет один человек, важно их определить для структурирования работы:
*   **Разработчик (Developer):**
    *   *Задачи:* Написание и отладка кода, реализация всех модулей конвейера, создание CLI-интерфейса.
    *   *Ответственность:* Техническое качество и работоспособность программного продукта.
*   **Системный аналитик (System Analyst):**
    *   *Задачи:* Изучение бизнес-процессов, анализ нормативной документации (ГОСТ), формулирование технических требований, подготовка отчета.
    *   *Ответственность:* Соответствие продукта решаемой бизнес-задаче.
*   **Инженер по тестированию (QA Engineer):**
    *   *Задачи:* Подготовка набора тестовых данных (изображений разного качества), проведение ручного и автоматизированного тестирования, документирование дефектов.
    *   *Ответственность:* Стабильность работы продукта и выявление узких мест.

**План коммуникаций:**

1.  **Владелец продукта -> Спонсор проекта:** Регулярные (например, еженедельные) демонстрации прототипа и отчеты о достигнутом прогрессе для подтверждения движения в верном направлении.
2.  **Владелец продукта -> Конечные пользователи:** Проведение интервью для сбора требований (на начальном этапе) и демонстрация промежуточных версий для получения обратной связи по удобству использования и формату данных.
3.  **Владелец продукта -> Руководитель практики от университета:** Обсуждение технических решений, архитектуры, сложностей реализации и содержания отчетной документации.

Такой подход позволяет структурировать инициативный проект в понятных для управления терминах, даже при отсутствии формальной проектной команды.

#### 1.3. Безопасные условия жизнедеятельности

Перед началом работ на предприятии был пройден обязательный инструктаж по технике безопасности, охране труда и пожарной безопасности. Основная работа велась за персональным компьютером. При работе соблюдались следующие правила:
-   Организация эргономичного рабочего места (правильная высота стула, освещение, расположение монитора).
-   Соблюдение режима труда и отдыха, выполнение регулярных перерывов для предотвращения зрительного и общего утомления.
-   Использование лицензионного или открытого программного обеспечения для избежания юридических и вирусных рисков.

С точки зрения безопасности самого продукта, были четко определены его границы, что является ключевым аспектом инженерной этики и соблюдения законодательства. Система **не обрабатывает и не интерпретирует геометрию чертежей** (линии, размеры, схемы) и **не работает с актуальной, находящейся в разработке проектной документацией**, доступ к которой ограничен по соображениям **государственной и коммерческой тайны**. Это полностью исключает риск внесения критических ошибок, влияющих на безопасность проектируемых или эксплуатируемых объектов, а также нарушений режима секретности. Продукт оперирует исключительно метаданными архивных документов, где потенциальная ошибка не несет катастрофических последствий и легко исправляется на этапе верификации. Взаимодействие с реальными данными предприятия, даже архивными, требует подписания соответствующих соглашений о неразглашении конфиденциальной информации (NDA).

Для разработки и тестирования прототипа использовались учебные и демонстрационные чертежи, взятые из открытых интернет-источников. Так как данные изображения не имели явной лицензии, запрещающей их использование в образовательных или исследовательских целях, они были применены исключительно для отладки алгоритмов и демонстрации работоспособности прототипа. Данный подход позволил полностью избежать работы с реальными архивными данными организации на этапе разработки.

### Раздел 2. Практическая часть

#### 2.1. Аналитический обзор и постановка задачи

**Научно-техническое обоснование.**
Проблема, решаемая проектом — системный разрыв между физической оцифровкой архивов и их информационной переработкой. Ручная обработка сканов, включающая поиск штампа и перепечатку данных, медленна, дорога и подвержена ошибкам. Автоматизация этого процесса с помощью технологий компьютерного зрения (CV) и оптического распознавания символов (OCR) является современным и эффективным решением.

**Анализ нормативно-технической базы.**
Основой для алгоритмов локализации и сегментации штампа послужили стандарты Единой системы конструкторской документации (ЕСКД) и Системы проектной документации для строительства (СПДС). Ключевыми документами являются:
-   **ГОСТ 2.104-68 (отменен, но актуален для старых архивов)**: Устанавливал формы основных надписей в СССР.
-   **ГОСТ 21.101-97 (отменен)**: Определял формы штампов для строительства до 2021 года.
-   **ГОСТ 21.101-2020 (действующий)**: Устанавливает современные требования к оформлению.

Конвейер `gost-ocr` спроектирован с учетом вариативности штампов. В частности, алгоритм локализации проверяет контуры на соответствие пропорциям основных форм (например, форма 3/5 с размерами 185×55 мм и форма 4/6 с размерами 185×15 мм), что позволяет обрабатывать различные типы документов.

**Анализ аналогов.**
Существуют коммерческие системы OCR (например, ABBYY FineReader), способные распознавать текст на документах. Однако они часто являются универсальными и могут требовать дополнительной настройки для специфической задачи сегментации полей ГОСТ-штампа. Также существуют open-source движки OCR (`Tesseract`, `EasyOCR`), которые предоставляют базовые возможности распознавания, но не решают задачу локализации и структурирования данных из коробки. Проект `gost-ocr` занимает нишу специализированного инструмента, объединяющего локализацию и распознавание именно для ГОСТ-чертежей.

**Формирование технического задания.**
На основе проведенного анализа было сформировано предварительное техническое задание, ключевые пункты которого легли в основу технической спецификации проекта (см. Приложение А).

#### 2.2. Проектирование информационной системы

**Архитектура системы.**
Проект `gost-ocr` реализован как CLI-утилита на Python и следует трехступенчатой архитектуре конвейера:

1.  **Предобработка (`preprocessing.py`)**: На этом этапе исходное изображение загружается, опционально поворачивается на 90/180/270 градусов (`--flip`) для поиска штампа в любой ориентации, и подвергается коррекции перекоса (deskew). Также возможно выделение "области интереса" (ROI) для ускорения поиска (по умолчанию — правый нижний квадрант изображения).
2.  **Локализация (`localization.py`)**: В области интереса производится поиск контуров. С помощью анализа иерархии (`RETR_TREE`) и геометрических свойств (соотношение сторон, количество вершин) находится внешний контур штампа. Алгоритм валидирует найденные прямоугольники на соответствие пропорциям стандартных форм ГОСТ.
3.  **Извлечение (`extraction.py`)**: Внутри найденной рамки штампа используется движок `EasyOCR` для распознавания всего текста. Результаты (текстовые блоки, их координаты и уверенность) вместе с координатами штампа сохраняются в JSON-файл.

**Технологический стек.**
-   **Язык программирования**: Python 3.12.
-   **Компьютерное зрение**: `opencv-python` — для всех операций с изображениями: загрузка, преобразования, поиск контуров, коррекция перекоса.
-   **OCR-движок**: `EasyOCR` — для распознавания текста. Выбран за простоту интеграции и хорошее качество распознавания русского языка.
-   **CLI-интерфейс**: `Typer` — для создания удобного и интуитивно понятного интерфейса командной строки.
-   **Математические операции**: `NumPy` — для эффективной работы с массивами изображений.
-   **Управление зависимостями**: `uv` и `pyproject.toml` — для быстрой и декларативной установки зависимостей.

**Структура данных.**
Основным форматом вывода является JSON, который содержит следующую информацию:
-   `source_image_path`: Путь к исходному файлу.
-   `stamp_bounding_box`: Координаты найденной рамки штампа.
-   `text_blocks`: Массив объектов, где каждый объект представляет распознанный блок текста с его содержимым (`text`), уверенностью (`confidence`) и координатами (`box`).
-   `full_text`: Весь распознанный текст в виде одной строки.

Такая структура позволяет легко интегрировать результаты работы конвейера с другими системами (например, СЭД) или использовать их для дальнейшего анализа.

#### 2.3. Разработка и тестирование

**Разработка.**
Код проекта был организован в виде пакета `gost_ocr` с четким разделением логики по модулям, что соответствует спроектированной архитектуре:
-   `cli.py`: Точка входа, обработка аргументов командной строки.
-   `config.py`: Хранение констант (пути, параметры ГОСТ, пороги).
-   `preprocessing.py`: Функции предобработки.
-   `localization.py`: Функции локализации штампа.
-   `extraction.py`: Функции извлечения текста.

Такая структура упрощает поддержку, тестирование и дальнейшее расширение функциональности (например, добавление модуля сегментации штампа на отдельные ячейки).

**Тестирование.**
Для проверки корректности работы были применены следующие подходы:
-   **Модульное тестирование (ручное)**: Каждый модуль проверялся отдельно. Например, для модуля `localization.py` использовался флаг `--draw-all`, который визуализирует все найденные прямоугольные контуры, позволяя оценить корректность работы алгоритма фильтрации.
-   **Интеграционное тестирование**: Проверялась работа всего конвейера на наборе тестовых изображений, расположенных в `src/gost_ocr/tests/test_images/`. Эти изображения включают как качественные сканы, так и примеры с поворотом, перекосом и артефактами.
-   **Отладочный режим**: Использование флага `--debug` позволяет сохранять промежуточные результаты обработки (предобработанное изображение, ROI, изображение с нарисованными контурами) в папку `debug/`. Это ключевой инструмент для визуальной проверки и отладки каждого шага конвейера.

Целью тестирования на данном этапе была не количественная оценка точности (precision/recall), а качественная проверка работоспособности всех компонентов системы и их устойчивости к типичным дефектам сканирования.

## Приложение А. Техническая спецификация проекта GOST-OCR

**Исходный код проекта доступен по ссылке:** [https://github.com/Sapar-hub/gost-ocr.git](https://github.com/Sapar-hub/gost-ocr.git)

### 1. Ключевая функциональность

Основная цель проекта — разработать приложение на Python для оптического распознавания символов (OCR) на технических чертежах, оформленных по стандартам ГОСТ. Задача заключается в автоматическом извлечении структурированных данных из основной надписи (штампа).

### 2. Архитектура и поток данных

Приложение построено на основе двухэтапной конвейерной архитектуры (третий этап, извлечение, запланирован для будущей реализации).

**Поток данных:**
1.  **Вход:** Графические файлы (PNG, JPG) из указанной папки или по пути к одному файлу.
2.  **Предобработка:** Коррекция перекоса (deskew), опциональные повороты и выделение области интереса (ROI).
3.  **Локализация:** Детектирование контуров с анализом иерархии для нахождения рамки штампа.
4.  **Извлечение:** Распознавание текста выполняется с помощью `EasyOCR`.

### 3. Ввод / Вывод

*   **Вход:**
    *   Формат: Графические файлы (PNG, JPG, JPEG).
    *   Источник: Путь к локальному файлу или директории.
    *   Путь по умолчанию: `src/gost_ocr/tests/test_images/`.
*   **Вывод:**
    *   **Основной:** Координаты рамки штампа и метаданные в формате JSON.
    *   **Отладочный:** Промежуточные изображения, сохраняемые в директории `debug/preprocessing/` и `debug/localization/`.

### 4. Ключевые модули

*   `cli.py`: Интерфейс командной строки на базе Typer с командами: `preprocess`, `localize`, `pipeline`.
*   `preprocessing.py`: Загрузка изображений, коррекция перекоса, повороты, выделение ROI.
*   `localization.py`: Детектирование контуров с иерархией `RETR_TREE`, валидация по пропорциям ГОСТ.
*   `config.py`: Константы конфигурации (размеры по ГОСТ, пороги, пути).

### 5. Этап 1: Предобработка

#### 5.1. Загрузка изображений

*   Загружает файлы PNG/JPG/JPEG из указанного пути.
*   Поддерживает как одиночные файлы, так и директории.
*   Выводит в консоль количество загруженных файлов.

#### 5.2. Коррекция перекоса (Deskew)

*   Использует преобразование Хафа (`cv2.HoughLinesP`) для определения доминирующих углов наклона линий.
*   Фильтрует линии, близкие к горизонтальным (±45°).
*   Вычисляет пиковый угол по гистограмме и поворачивает изображение для исправления перекоса.

#### 5.3. Повороты

*   По умолчанию отключены.
*   Включаются через флаг `--flip` / `-f` в CLI.
*   Применяет повороты: 0°, 90°, 180°, 270° (создает 4 версии для каждого изображения).

#### 5.4. Выделение области интереса (ROI)

*   Выделяет 1/4 области изображения (50% ширины × 50% высоты).
*   Позиция по умолчанию: правый нижний угол.
*   Настраивается через флаг `--roi`: `bottom_right`, `bottom_left`, `top_right`, `top_left`.

#### 5.5. Отладочный вывод

При установленном флаге `--debug` / `-d`:
*   Сохраняет в `debug/preprocessing/`.
*   Файлы: `{name}_preprocessed.png`, `{name}_roi.png`.

### 6. Этап 2: Локализация

#### 6.1. Детектирование контуров

*   Преобразует ROI в оттенки серого.
*   Применяет адаптивную бинаризацию (`cv2.ADAPTIVE_THRESH_GAUSSIAN_C`).
*   Выполняет морфологическое закрытие с ядром 25×25 для формирования сплошных областей.
*   Использует `cv2.findContours` с режимом `RETR_TREE` для получения иерархии.

#### 6.2. Анализ иерархии

*   Режим `RETR_TREE` возвращает отношения родитель-потомок.
*   Алгоритм ищет контуры, которые являются родительскими (имеют дочерние контуры), но сами не имеют родителя.
*   Это позволяет корректно идентифицировать внешнюю рамку штампа, игнорируя внутренние ячейки.

#### 6.3. Фильтрация прямоугольных контуров

*   Использует `cv2.approxPolyDP` для проверки, имеет ли контур 4 вершины.
*   Фильтрует по минимальной площади, ширине и высоте.

#### 6.4. Валидация по пропорциям ГОСТ

Проверяет соотношение сторон на соответствие стандартам ГОСТ:
*   **Форма 3/5**: 185×55мм → соотношение сторон ≈ 3.36
*   **Форма 4/6**: 185×15мм → соотношение сторон ≈ 12.33
*   Допуск: ±0.5

#### 6.5. Режим отрисовки всех контуров

При установленном флаге `--draw-all`:
*   Пропускает валидацию по пропорциям ГОСТ.
*   Находит все прямоугольные контуры.
*   Отрисовывает все контуры на отладочном изображении.
*   Выводит количество найденных контуров вместо информации о штампе.

#### 6.6. Отладочный вывод

При установленном флаге `--debug` / `-d`:
*   Сохраняет в `debug/localization/`.
*   Файл: `{name}_localization.png` с отрисованными рамками.

### 7. Команды CLI

```
gost-ocr --help
gost-ocr preprocess [PATH] [--flip] [--roi POSITION] [--debug]
gost-ocr localize [PATH] [--flip] [--draw-all] [--roi POSITION] [--debug]
gost-ocr pipeline [PATH] [OPTIONS]  # псевдоним для localize
```

**Опции:**
*   `--flip` / `-f`: Применить все повороты (0/90/180/270°).
*   `--draw-all`: Показать все прямоугольные контуры (пропустить валидацию ГОСТ).
*   `--roi`: Позиция ROI (`bottom_right`, `bottom_left`, `top_right`, `top_left`).
*   `--debug` / `-d`: Сохранить промежуточные результаты.
