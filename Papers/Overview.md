## Введение в проектный отчет: Проблематика автоматизации обработки метаданных архивных чертежей в проектных институтах

### 1. Контекст проблемы: цифровизация исторических архивов в инженерной документации

Проектные институты России, включая АО «Метрогипротранс» (основан в 1933 г.), накопили многодесятилетние архивы проектной документации на бумажных носителях. Согласно Федеральному закону № 125-ФЗ «Об архивном деле в РФ» и программе цифровизации госсектора, эти архивы подлежат оцифровке с сохранением юридической значимости документов.

Однако процесс оцифровки сталкивается с системной проблемой: **сканирование преобразует документ в изображение, но не в структурированные данные**. В результате:
- Сканы архивных чертежей сохраняются под техническими именами (`scan_001.tif`, `IMG_20240208_142355.jpg`);
- Поиск конкретного листа проекта требует ручного просмотра тысяч файлов;
- Интеграция с электронными системами документооборота (СЭД) невозможна без ручного ввода метаданных.

Эта рутина ложится на младший технический персонал (архивариусов, техников-проектировщиков), отвлекая их от задач с добавленной стоимостью. Согласно исследованиям в области цифровизации строительной отрасли (Вирцев, 2024; Волкова, 2022), до 30% времени сотрудников архивных подразделений уходит на ручную обработку метаданных сканированных документов.

### 2. Специфика проектной документации метрополитена

Документация Метрогипротранса имеет особенности, влияющие на автоматизацию:

| Фактор                            | Влияние на обработку                                                                                                                                                                  |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Длительность жизненного цикла** | Проекты 1960–1980-х гг. до сих пор используются при реконструкции (станция «Комсомольская», 1935 г.). Архивные сканы имеют низкое разрешение (150–200 dpi), пятна, деформации бумаги. |
| **Многослойность обозначений**    | Шифр проекта содержит: префикс «М» (метрополитен), номер проекта, обозначение раздела (АР/КР/ОВ/ТМ). В СССР использовались иные обозначения (АС вместо АР+КР).                        |
| **Строгая регламентация**         | Все документы оформляются по ГОСТ 21.101 (основные надписи), ГОСТ 21.501 (правила выполнения рабочей документации). Отклонения от стандарта возможны в архивных копиях.               |
| **Юридическая критичность**       | Ошибка в метаданных (неверный шифр) не угрожает безопасности строительства, но нарушает трассируемость документа в системе управления проектами — что критично для госзаказчика.      |

> **Важно**: Работа ограничена **метаданными** (шифр, номер листа, стадия), а не геометрией чертежей. Это принципиальное разделение зон ответственности: распознавание размеров или арматуры требует сертифицированного ПО и аттестации специалиста (СП 11-104-96), тогда как обработка штампа относится к вспомогательным операциям.

### 3. Описание рутинной операции, подлежащей автоматизации

Типовой сценарий ручной обработки архивного скана:

1. Архивариус получает папку с 500 сканами станции метро 1970-х гг. (`scan_001.jpg` … `scan_500.jpg`).
2. Для каждого файла:
   - Открывает изображение в просмотрщике;
   - Визуально ищет основную надпись (штамп) в углу листа;
   - Считывает шифр проекта (напр., `М-1234-АР`);
   - Считывает номер листа (напр., `07/24`);
   - Переименовывает файл: `М-1234-АР_лист_07.jpg`;
   - Вносит данные в реестр Excel.
3. На 500 листов уходит 3–5 рабочих дней при концентрации внимания.

**Боль**: задача монотонна, подвержена человеческим ошибкам (перепутать `07/24` и `17/24`), не масштабируется при массовой оцифровке.

### 4. Технические задачи конвейера обработки

Цель проекта — разработать прототип конвейера для автоматического извлечения метаданных из штампов архивных чертежей. Конвейер реализует трехступенчатую архитектуру:

#### Этап 1: Локализация штампа (Object Detection without Training)

**Проблема**: Штамп по ГОСТ 21.101-97 размещается в правом нижнем углу поля чертежа, но в архивных сканах возможны:
- Смещение из-за неровного сканирования;
- Использование формы 6 (спецификации) с размещением в левом нижнем углу;
- Поврот листа (альбомная ориентация);
- Отсутствие рамки чертежа на копиях «синьках».

**Решение**:
```python
# Многоугольный поиск по признакам прямоугольной рамки
def locate_stamp_candidates(image):
    candidates = []
    # Поиск в 4 углах с приоритетом правого нижнего
    for zone in [(0.85, 0.70), (0.85, 0.0), (0.0, 0.70), (0.0, 0.0)]:
        roi = extract_corner_zone(image, zone)
        contours = find_rectangular_contours(roi)  # OpenCV: Canny + findContours
        if contours and is_stamp_proportions(contours[0]):  # соотношение ~3.4:1 (185х55 мм)
            candidates.append((zone, contours[0]))
    return sorted(candidates, key=lambda x: priority_score(x[0]))  # правый нижний = приоритет 1
```
*Инженерный нюанс*: Отказ от обучения нейросети (YOLO) обоснован отсутствием размеченных данных и необходимостью интерпретируемости. Контурный анализ по ГОСТ-пропорциям обеспечивает воспроизводимость.

#### Этап 2: Классификация типа документа

**Проблема**: Формат штампа зависит от типа документа и его положения в пакете:
- **Форма 3/5**: Основные листы чертежей и текстовых документов. Имеют размер **185х55 мм** (соотношение сторон ~3.4:1).
- **Форма 4/6**: Последующие листы и спецификации. Имеют размер **185х15 мм** (соотношение сторон ~12.3:1). Спецификации часто располагаются в левом углу.

**Решение**:
```python
# Классификация на основе пропорций, количества ячеек и расположения
def classify_document_type(stamp_image, zone):
    h, w = stamp_image.shape[:2]
    aspect_ratio = w / h
    vertical_proj = np.sum(stamp_image, axis=0)
    cell_count = count_peaks(vertical_proj)
    
    # Константы: FORM_3_5_ASPECT_RATIO = ~3.36, FORM_4_6_ASPECT_RATIO = ~12.33
    if cell_count == 6 and abs(aspect_ratio - FORM_3_5_ASPECT_RATIO) < ASPECT_RATIO_TOLERANCE:
        return "FORM_3_5"  # Основной лист (чертеж/текст)
    elif cell_count == 4 and abs(aspect_ratio - FORM_4_6_ASPECT_RATIO) < ASPECT_RATIO_TOLERANCE:
        if is_left_aligned(zone):
            return "FORM_6"  # Спецификация
        else:
            return "FORM_4_6" # Последующий лист
    else:
        return "UNKNOWN"
```
*Инженерный нюанс*: Классификация основана на геометрии сетки штампа (фиксированной по ГОСТ), а не на содержимом — это исключает ошибки из-за плохого качества текста.

#### Этап 3: Извлечение и валидация метаданных

**Проблема**: Текст в штампе содержит:
- Структурированные поля (шифр, лист) с предсказуемым форматом;
- Неструктурированные поля (название объекта) с вариативностью;
- Рукописные пометки (резолюции), не подлежащие автоматической обработке.

**Решение**:
```python
def extract_metadata(stamp_image, doc_type):
    # Сегментация ячеек по координатам ГОСТ
    cells = segment_stamp_cells(stamp_image, doc_type)  
    
    # Извлечение с валидацией
    metadata = {}
    metadata['project_code'] = validate_with_regex(
        ocr(cells['code']), 
        r'[ММ]-\d{3,4}(-[А-Я]{2,3})?'  # допускаем отсутствие раздела
    )
    metadata['sheet'] = validate_with_regex(
        ocr(cells['sheet']), 
        r'(\d{1,2})/(\d{1,2})'  # 07/24
    )
    
    # Confidence scoring для инженерной этики
    metadata['confidence'] = min(
        tesseract_confidence(cells['code']),
        geometry_validation(cells['code'])  # проверка расположения ячейки
    )
    
    # Пометка на ручную проверку при низком доверии
    if metadata['confidence'] < 0.85:
        metadata['requires_review'] = True
    
    return metadata
```
*Инженерный нюанс*: Система **не заменяет человека**, а фильтрует документы:
- Высокая уверенность (>0.95) → автоматическое переименование;
- Средняя (0.85–0.95) → предзаполнение формы для оператора;
- Низкая (<0.85) → пометка «требует ручной проверки».

### 5. Границы применимости и этические ограничения

Прототип разрабатывается с соблюдением принципов инженерной ответственности:

| Ограничение                    | Обоснование                                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| **Только метаданные**          | Геометрия чертежа (линии, размеры) не обрабатывается — это зона ответственности сертифицированного инженера. |
| **Только архивные документы**  | Работа ведется на учебных данных (публичные чертежи), а не на актуальных проектах.                           |
| **Требуется ручная валидация** | Система выдает `confidence score`; окончательное решение принимает оператор.                                 |
| **Не для BIM-интеграции**      | Извлеченные данные не используются для генерации 3D-моделей без ручной проверки инженером.                   |
### «Перспективы внедрения»

> «Прототип реализован как автономная CLI-утилита, не зависящая от конкретной СЭД. Интеграция с 1С:Предприятие возможна тремя способами:
> 
> 1. **Вызов через командную строку** — требует настройки прав на сервере 1С и организации файлового обмена (входная папка → выходной XLSX). Подходит для локальных инсталляций СЭД.
> 2. **Веб-сервис на Python** — современный подход с изоляцией компонентов. Требует развёртывания отдельного HTTP-сервера и настройки межсистемной аутентификации.
> 3. **Файловая очередь** — минимально инвазивный способ без изменения кода 1С. Подходит для пилотного внедрения.
> 
> Прямой вызов Python-скрипта из кода 1С невозможен по архитектурным ограничениям платформы. Любая интеграция требует промежуточного слоя (файлы, HTTP) и согласования с ИБ-службой организации.»
### Заключение раздела

Данный проект не претендует на создание промышленного решения для критически важных операций. Его цель — демонстрация технической возможности автоматизации **вспомогательных операций** с метаданными при строгом соблюдении границ ответственности. Прототип решает реальную рутинную задачу (обработка штампов), характерную для проектных институтов при оцифровке архивов, и может служить основой для дальнейшей разработки после согласования с инженерами-практиками и юридической службой организации.
